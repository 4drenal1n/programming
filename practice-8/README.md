Контрольные вопросы
 1. Зачем нужны include guards?

    Include guards (или header guards) предотвращают многократное включение одного и того же      
    заголовочного файла в одном компилируемом модуле.

     1 #ifndef MODULE_H
     2 #define MODULE_H
     3 // Объявления функций, структур и т.д.
     4 #endif // MODULE_H

    Без include guards:
     - При включении заголовка в несколько файлов может возникнуть ошибка повторного определения  
     - Увеличивается время компиляции
     - Возможны конфликты имён

    С альтернативой `#pragma once`:

     1 #pragma once
     2 // Работает быстрее, но менее переносим (не все компиляторы поддерживают)

    
2. Чем объявление отличается от определения?


    ┌──────────────────────────────────────┬──────────────────────────────────────────┐
    │ Объявление (Declaration)             │ Определение (Definition)                 │
    ├──────────────────────────────────────┼──────────────────────────────────────────┤
    │ Сообщает компилятору об имени и типе │ Выделяет память и создаёт объект/функцию │
    │ Может быть много раз                 │ Должно быть только одно (правило ODR)    │
    │ Не выделяет память                   │ Выделяет память                          │
    │ Пример: extern int x;                │ Пример: int x = 5;                       │
    │ Пример: void func(int);              │ Пример: void func(int) { ... }           │
    └──────────────────────────────────────┴──────────────────────────────────────────┘


    Для функций:                                                                                  
                                                                                                  
     1 // Объявление (в заголовке)
     2 void my_function(int x);
     3
     4 // Определение (в исходном файле)
     5 void my_function(int x) {
     6     // тело функции
     7 }

    Для переменных:

     1 // Объявление (в заголовке, с extern)
     2 extern int global_var;
     3
     4 // Определение (в одном .c файле)
     5 int global_var = 42;

    

3. Что такое статическая линковка?

    Статическая линковка — это процесс включения кода библиотеки непосредственно в исполняемый    
    файл на этапе компиляции.

    Характеристики:
     - Библиотека копируется в EXE-файл целиком (или нужные части)
     - Исполняемый файл становится больше
     - Не требует наличия библиотеки при запуске программы
     - Компиляция происходит дольше

    В Makefile:

     1 # Создание статической библиотеки
     2 $(LIB): $(LIB_OBJS)
     3     ar rcs $@ $^
     4
     5 # Линковка со статической библиотекой
     6 $(TARGET): $(OBJ_DIR)/main.o $(LIB)
     7     $(CC) $< -L$(BIN_DIR) -lmathlib -o $@

    В Meson:

     1 mathlib = static_library('mathlib', lib_src, ...)
     2 executable('demo', 'src/main.c', link_with: mathlib)

    Преимущества:
     - Портативность (один файл)
     - Нет зависимостей во время выполнения

    Недостатки:
     - Больший размер EXE
     - Обновление библиотеки требует перекомпиляции

    
4. Как работает переменная $^ в Makefile?

    $^ — это автоматическая переменная Make, которая содержит все зависимости цели, разделённые   
    пробелами.

    Пример:

     1 program: main.o utils.o math.o
     2     $(CC) $^ -o program

    Развернётся в:

     1 gcc main.o utils.o math.o -o program

    Другие автоматические переменные:


    ┌────────────┬─────────────────────────────────┐
    │ Переменная │ Значение                        │
    ├────────────┼─────────────────────────────────┤
    │ $@         │ Имя цели                        │
    │ $<         │ Первая зависимость              │
    │ $^         │ Все зависимости                 │
    │ $?         │ Зависимости, которые новее цели │
    │ $*         │ Имя файла без расширения        │
    └────────────┴─────────────────────────────────┘


    Пример использования:

     1 # $@ = libmathlib.a, $^ = все .o файлы
     2 $(LIB): $(LIB_OBJS)
     3     ar rcs $@ $^
     4 
     5 # $@ = obj/main.o, $< = src/main.c
     6 $(OBJ_DIR)/%.o: $(SRC_DIR)/%.c
     7     $(CC) $(CFLAGS) -c $< -o $@

    
5. В чём преимущества Meson перед Make?


    ┌─────────────────────────────────────────────────────┬───────────────────────────────────┐
    │ Meson                                               │ Make                              │
    ├─────────────────────────────────────────────────────┼───────────────────────────────────┤
    │ Кроссплатформенность — работает одинаково на Win... │ Зависит от системы (GNU Make, ... │
    │ Автоматическое обнаружение зависимостей             │ Нужно вручную указывать           │
    │ Встроенная поддержка тестов (meson test)            │ Нужно писать правила вручную      │
    │ Конфигурация сборки (debug/release) из коробки      │ Нужно настраивать отдельно        │
    │ Генерация проектов для IDE (Visual Studio, Xcode... │ Нет                               │
    │ Проще синтаксис (Python-подобный)                   │ Сложный синтаксис с табуляцией    │
    │ Автоматическое отслеживание зависимостей            │ Нужно указывать .d файлы          │
    │ Встроенная установка (meson install)                │ Нужно писать правила              │
    └─────────────────────────────────────────────────────┴───────────────────────────────────┘


    Пример сравнения:

    Makefile:

      1 CC = gcc
      2 CFLAGS = -Wall -I include
      3 SRCS = $(wildcard src/*.c)
      4 OBJS = $(SRCS:.c=.o)
      5
      6 program: $(OBJS)
      7     $(CC) $(OBJS) -o $@
      8
      9 %.o: %.c
     10     $(CC) $(CFLAGS) -c $< -o $@
     11
     12 clean:
     13     rm -f $(OBJS) program

    meson.build:

     1 project('myproject', 'c')
     2 inc = include_directories('include')
     3 sources = files('src/main.c', 'src/utils.c')
     4 executable('program', sources, include_directories: inc)

    Недостатки Meson:
     - Требует Python и Ninja
     - Меньше контроля над деталями сборки
     - Нужно изучать новый синтаксис
